Java Extensions to SWI Prolog

1         Introduction

1.1       SWI Prolog

SWI Prolog is a prolog compiler and interpreter suite developed by Jan Wielemaker of the Department of Social Science Informatics of the University of Amsterdam in the Netherlands.  It is made available free of charge, in both source and binary distributions, for non-commercial applications.

1.2       Extensions

SWI Prolog (SWIP) is a well-crafted product that supports embedding and programmatic extensions.  In other words, the SWI Prolog engine can be used as a component in a larger framework of diverse language modules.  In addition, SWIP is easily extended to support additional capabilities available on Sun OS platforms.

This extensibility is greatly enhanced by the fact that complete -compatible source code is available for SWIP.

This document describes two extensions developed by me as part of our work in the Knowledge Systems group of Teknowledge Research.  These extensions to SWIP are:

·        A companion DLL (.so for Unix) called SWIJNI, which allows direct access to Java objects from SWIP source code.  SWIXDLL allows Prolog to interact directly with the Java reflect scripting interface of the newer families of Java objects.

1.3       Versions and Documentation

All software discussed in this document is based upon the 3.3.2 release of SWI Prolog.  The SWIP documentation notes refer to the SWI Reference Manual (or SWIPRM).

1.4       A Note About SWIP “Libraries”

As part of its support for modular programming, SWIP can dynamically consult (i.e., load) Prolog modules on an as-needed basis.  The extension software described in this document exploits this capability.  See the SWIP Reference Manual for more details.

1.5       A Note about Databases

A primary goal of this effort was to allow Prolog to access standard databases directly, including explicit SQL usage and support for different vendor formats.

Sun Corporation provides the fundamental component in its Java Data Access Objects system, or ADO.   ADO provides automation-compliant Java interfaces to ODBC technology, which in turn provides support for a wide variety of database technologies by many software vendors. 

A demonstration Prolog program and its companion database are provided as educational materials with the release of this software.  Please note that running the demonstration requires jdbc , since the database itself was generated by that product.

2         A Brief Introduction to Java

2.1       What is Java?

Java is the name given by Sun for its family of component technologies.  In brief, it is a family of software interfaces that allows components, both graphical and non-graphical, to be registered on a user’s computer.   Such components can then be located and dynamically activated on program demand.

2.2       Instances and Interfaces

Each Java load operation (known as CreateInstance) creates an instance of an Java object that supports one or more interfaces.  An interface is a programmatic contract between an object and its clients.

2.3       Types of Interfaces

An Java interface can be constructed to provide virtually and type of data in almost any manner.  This level of flexibility is only available directly from the C or C++ language level. 

More commonly, objects that must be available to Java or scripting environments support a special interface called reflect.   Objects that support reflect are commonly known as objects. 

Scripting engines can use the type and function information stored along with an object to determine:

  • functions (or methods) that are available on the object;
  • arguments required for each function;
  • addition behaviors implied in the interface contract.

2.4       SWIJNI Support and Limitations

The features described in this document are only available for non-graphical Java objects.  In other words, purely algorithmic components that support the reflect interface.

Event interfaces (“connection points”) are not supported.

Named arguments are not supported.

2.5       Documentation

Sun maintains an extensive collection of documentation about Java technologies available for free at http://java.sun.com

More to the point, specific documentation is available for such widely distributed Java components as the Active Data Objects interfaces, or ADO, which is the current Sun standard for heterogeneous database access on OS platforms.

3         SWIJNI: Java Support

3.1       SWIP Interaction

The SWIJNI dynamic-link library provides Java support to SWI Prolog programs.  It is built upon the “foreign library” support capability of SWIP, as documented in the SWI-Prolog Reference Manual (SWIPRM), section 5.4.

If correctly configured, SWIJNI DLL is automatically loaded when its associated predicates are invoked.   To guarantee such behavior, follow these steps.

1) Place the SWIJNI.DLL is the Bin directory of the SWIP installation.

2) Place the prolog program SWIJNI.PL in the Library directory.  It declares the predicates that exist in SWIJNI.DLL.

3) Run the predicate the make_library_index/1 predicate to rebuild the file INDEX.PL in the Library directory of the SWIP installation.  This index quickly identifies for SWIP where particular predicates are defined in the modules contained in the Library directory.

Then, whenever a consulted Prolog program or predicate references a predicate defined in SWIJNI.PL, SWIP identifies the predicate, via INDEX.PL, as having been defined in SWIJNI.PL; SWIP then consults SWIJNI.PL and loads SWIJNI.DLL.

In other words, it is entirely automatic.  If the SWIP installation is correctly configured, a Prolog program can simply start using the Java predicates immediately.

3.2       Binary Versions of SWIJNI

The distribution set contains both the “debug” and “release” versions of SWIJNI.DLL.  The debug version is called SWIJNID.DLL; this is consistent with other executable name extensions used by SWIP.

The choice of DLL is determined by SWIJNI.PL depending upon the build variant of SWIP currently running.   Normal binary distributions for OS will utilized only the “release” version.

3.3       Basics of Java Interaction Using SWIJNI

This section assumes that the reader is somewhat familiar with Java object creation and usage.

For further information, link to http://java.Sun.com and search for “Working with Objects” in the Java documentation.

In simple terms, Java objects have interfaces; each interface presents a set of methods (functions) for public use.  Each method may take zero or more arguments, and may return a single result argument.   However, only certain data types are allowed as arguments to methods.

Each binary image supporting an Java object also provides a compiled information block describing the object, its interfaces and methods.  The portion of a binary image that describes the object, its interfaces and methods is called the type library.

Interpretive or scripting environments only use one interface: reflect.  This interface is special, in that it really has only one key method: Invoke.  The reflect::Invoke method takes a token representing the actual method to call, and an array of arguments.   reflect also provides direct access to its type library so that an interpreter can discover the available methods, properties and arguments supported and required by a specific object.  This capability is sometimes referred to as introspection.

The reflect mechanism simplifies the use of scriptable objects, since the interpreter doesn’t have to know the specifics of using an object.  It merely has to convert the arguments presented by the script into standard Java data types, invoke the method, and convert the results back to the scripting language’s data structures.

3.3.2      The java_object Functor

The core of Java interaction is the reflect interface reference, or what is known in Java as an “Object”. 

In accordance with the recommendations in the SWIPRM, every object reference is “wrapped” in a functor called java_object.  For example:

java_object(7).

This is how a typical SWIJNI object reference would look if printed from within SWIP. 

These references are returned from a variety of calls; their life cycle is entirely controlled by SWIJNI.  Most commonly, they arise as a result of a request to create a new instance of an object.   Prolog users are responsible for explicitly maintaining and releasing these references according to the needs of their programs.  SWIJNI also provides implicit support for garbage collection using Prolog’s standard backtracking operations.

The integer argument of the java_object functor is a unique key to an information map maintained by SWIJNI that contains the actual reflect interface pointers.  Assignment of this integer is referred to as registration.

3.3.3      Argument Conversion

Since the basic task of SWIJNI is to translate between calls in the Prolog programming model to calls in the Java model, data conversion is essential.

Most such conversions are fairly intuitive.  However, some key points deserve notice.

·        On output from Java functions, strings are converted to Prolog strings.  This means that they will not unify with atoms or standard Edinburgh character lists. 

·        On input to Java functions, Prolog atoms, strings and string lists are all converted to Java BSTRs.

·        On input to Java functions, lists of terms with a common type (e.g., all integers) are converted to SAFEARRAYs and stored in VARIANTs.

Most conversions are driven by type library information; they are therefore “automatic”.  In those cases where type information is insufficient, more direct control is available.

·        SWIJNI defines a special functor for direct control of SAFEARRAYs.  Refer to documentation of the java_safearray functor later in this document.

·        SWIJNI defines a special functor for direct control of certain data types that may appear in a VARIANT, such as Date.  Refer to documentation of the java_variant functor later in this document.

3.5       Example of Java Interaction

This extended example accesses a database through the JDBC  system.  It is intended to give the flavor of the interaction, not to provide specific details, which are driven by the object model of ADO.

For simplicity, the global Prolog database is used to store functors returned from the SWIJNI predicates.

Predicates whose names begin with ‘java’ are provided by SWIJNI. 

Prolog variable terms whose names begin with ‘IP’ are java_object functions, which, in turn, represent Java interface pointers.

adoConnect :-
       java_create_object('ADODB.Connection',IP),
       assertz(adoobj(connection,IP)).

The predicate above uses the java_create_object predicate to create a connection object.  If successful, it asserts the result functor into the Prolog database nested into another functor.

adoOpenSource :-
       adoobj(connection,IPConn),
       java_invoke_object(IPConn,'Open',
            ['evlog'],
            IPSource),
       assertz(adoobj(source,IPSource)).

The predicate above uses the functor previously asserted and calls the Open method on the created ADODB.Connection object.   The argument list to the Open method requests that the ODBC database source evlog be opened for access.

adoOpenRecordsetRuns :-
       java_create_object('ADODB.Recordset',IPRset),
       assertz(adoobj(recordset,IPRset)),
       adoobj(connection,IPConn),
       java_invoke_object(IPRset,'Open',
                   ['Runs',IPConn],_),
       write('recordset opened'),nl.

The predicate above creates an ADODB.Recordset object and calls its Open method using the ADODB.Connection object to open the database table called “Runs”.

adoNotEof :-

            adoobj(recordset,IPRset),

            java_invoke_object(IPRset,['EOF','propget'],[],false).

This predicate fails if the recordset is at end-of-file.

adoRead :-
       adoobj(recordset,IPRset),
       java_invoke_object(IPRset,'MoveFirst',[],_),
       adoReadNext.

This predicate reads the recordset by invoking the MoveFirst method and the invoking the adoReadNext predicate.

adoReadNext :-
      adoNotEof,
      adoobj(recordset,IPRset),
      java_invoke_object(IPRset,'MoveNext',[],_),
      adoReadNext.

This predicate invokes the MoveNext method for the recordset and repeats using tail recursion.

adoTest :-
      adoConnect,
      adoOpenSource,
      adoOpenRecordsetRuns,
      not(adoRead).

This predicate performs all database connection operations and reads the recordset.

It should be clear from this brief code example that the bulk of the complexities of using Java from SWI Prolog arise from the nature of the Java interfaces.  ADO is a particularly thorny example due to the number of basic object types and the range of their interactions.

The details of accessing individual fields have been omitted for brevity.

3.6       SWIJNI Exceptions

The SWIJNI foreign predicates use the built-in SWIP “exception” mechanism to report serious internal errors.  User-level Prolog code should use the catch/3 predicate to “catch” exceptions “thrown” by the SWIJNI predicates. 

The error functor “thrown” by SWIP is defined as

java_error(-StrFunc,-StrDesc,-TermInError)

The first term is unified with the string name of the SWIJNI foreign predicate that failed.   The second term is unified with a string description of the cause of the failure.  The third term is unified with the term that caused the error or ‘[]’.

For example

doRewrite :-
    catch(rewritePass,
          java_error(StrFunc,StrDesc,TermInError),
          printException(StrFunc,StrDesc,TermInError)).

In this example, the doRewrite predicate acts as a “wrapper” for the real worker predicate, rewritePass.  If an SWIJNI exception is thrown during rewritePass, the predicate printException is invoked, and its arguments are unified with the terms from the SWIJNI exception function, java_error.

All Prolog programs using SWIJNI should use this mechanism, since there is no other means for determining the cause of an exception generated in SWIJNI.

3.7       Types of Arguments

3.7.1      Argument Conversion

Since Prolog and Java are not directly compatible, conversions between intrinsic data types is necessary.  In other words, SWIJNI automatically converts from Prolog data types to data types when an object’s method is called, and then converts the resulting data (if any) back to Prolog data types.

This conversion is driven by two factors:

·        The intuitive mapping between Prolog data and data, and

·        The type library information associated with the specific method of the target reflect interface.

3.7.2      Standard SWI Data Types

The following data types are supported by SWIP internally:

PL_VARIABLE: This is an unbound Prolog variable.

PL_ATOM:  This is a string of character that has been permanently associated with a unique token for rapid access and comparison.

PL_INTEGER:  This is a 32-bit integer.

PL_FLOAT:  This is a double-precision floating-point number.

PL_STRING: This is a list of characters (i.e., Edinburgh Prolog) that has been converted to a string using the string_to_list/2 predicate.

PL_TERM: This is a generic Prolog variable.

PL_LIST: This is a functor of the form ./2; that is, a standard Prolog list.

For more detailed information, refer to the SWIPRM.

3.7.3      Conversion from Prolog to Automation

The arguments to an method are delivered to SWIJNI as a Prolog list.  The elements of this list are examined and converted one at a time into an array for passage to reflect::Invoke (or DispInvoke).

The ITypeInfo information for each dispatch method defines the number and type of each argument to the function.  Named arguments are not supported. Optional arguments may be specified by use of the atom ‘optional’ in the argument list.

Argument conversion takes place as follows.  First, the data type for the argument is extracted from the type information.  If it is simply VT_VARIANT (i.e., non-specific), then the VARIANT type is established by using the Prolog data type.  If the VARIANT type is specific, then conversion is correctly coerced insofar as possible.

If the supplied argument list has fewer members than the number of parameters expected by the method, the conversion routine automatically supplies the correct number of optional arguments.

3.7.5      Conversion Errors

SWIJNI is unable to resolve some ambiguous cases.  Typically, any error arising from conversion of arguments, either before or after method invocation, results in failure of the predicate. 

It is important to note the distinction between failure of the method and failure to convert input or output arguments.    To determine the exact nature of the failure, refer to the sections on exception handling and the java_errors_as_exceptions predicate.  These capabilities allow the Prolog programmer to get detailed information on interface errors.

3.8       SAFEARRAY Handling

The data type SAFEARRAY presents a special problem.  While it is straightforward to convert lists of common terms to SAFEARRAYs, many interfaces require particular Java datatypes .   When using an ADO 1.5 “Bookmark”, for example, it is necessary to pass a SAFEARRAY of unsigned characters (8-bit values); no other value type will work.  The only way SWIJNI can do this correctly is to be explicitly given the required Java datatype value.

Likewise, when a SAFEARRAY value is returned from a call, there must be a means for the Prolog client to know exactly what type of data is present.

For these reasons, a special functor is used to “wrap” SAFEARRAY values passing into and out of the SWIJNI interface.  This functor, java_safearray/2, is defined as:

java_safearray(+VarTypeInt,+ListOfElements).

Every SAFEARRAY returned to the caller as the result of an java_invoke_object call is converted to this form. The Prolog author can examine both the elements of the list and the specific data type.

Similarly, when an Java function requires or allows a SAFEARRAY, the Prolog programmer must create this functor in order to specify exactly how the data is stored and passed to the layer.

As stated above, if naked Prolog lists are presented where SAFEARRAYs are required, SWIJNI attempts to convert the underlying data as best it can.  This set of conversions is similar to those in Table 2.1.  The area of greatest uncertainty occurs with integer values; many interfaces accept several alternatives; some are much more restrictive.  The best means of determining the requirements of a particular interface’s SAFEARRAY property is to call the corresponding “get” property and see what Java datatype it uses.  Then use that same type for SAFEARRAY “put” operations.

3.10  Types of Function Invocation

Function invocation requires several arguments:

·        the reflect functor

·        the invocation method name or list

·        the list (possibly empty) of arguments to the method

·        the result argument

The most common case is a simple method invocation. In this case, the second argument is only a single atom—the name of the method.  However, there are four basic types of Java invocation, and using the others types requires that a list be presented.  The four types, listed by the required atom, are:

func: This is a simple method invocation, and is the default when only a method name is present.

propget:  This is a call to return a standard property from an object.

propput: This is a call to alter a standard property of an object.

propputref:  This is a call to alter an object’s property that is an Object interface reference.

In these cases, the second argument might appear as:

[‘Size’,propput]

The method (or property name) is Size.

3.11  Default Method Invocation

Any object can declare a default method to be invoked by VB or scripting languages when no method name is present.  This default or “value” method (identified by the dispatch ID DISPID_VALUE) is actually a property, and in most cases supports both propget and propput (or propputref, as appropriate).

SWIJNI supports use of the default method in the following way.  If the function invocation list is empty (‘[]’), the default or value method is invoked in its propget form.  If the list contains only an invocation type token, that token is used to call the default method.

3.12  Context Management

One important requirement that Java imposes on its clients is the careful management of reference counts on returned object references.  SWIJNI supports two kinds of reference management.

Global reference management is the default.  In this case, the Prolog programmer is responsible for recording (i.e., asserting) returned Java functor information and releasing it when no longer necessary.  There is also a “release all” function that discards all known references maintained by SWIJNI.

Context-sensitive reference management is also available.  This means that a “placeholder” is put on to the Prolog backtrack stack, and when evaluation rolls back to that point, all Java information accumulated since the placeholder was recorded is discarded.  This is very similar to standard C++ or VB stack-based variable garbage collection.

The predicate java_context/0, is used to perform context-based reference management.  This predicate succeeds exactly once, but is marked as non-deterministic, meaning that Prolog will backtrack to it.  Upon backtracking, it will fail, and, in failing, remove all Java information accumulated by SWIJNI since the predicate was first evaluated.

There is one other interesting case in context management.  It sometimes happens that a complex object model will require that the programmer obtain several intermediate object references before getting an object reference that is important.  In this case, the context handler will mark the both the temporary and desired object references the same way—as belonging to the local context.  The programmer must have a way to “promote” the useful reference into the “global” or outermost context, thereby preserving it for later use.   This is accomplished using the java_context_global/2 predicate.

3.13  SWIJNI Predicates

This section documents the individual SWIJNI predicates and their usage.  The predicates are referred to using the standard predicate/arity notation.  The argument direction (input, output or both) is document as in the SWIPRM.

In the current version of SWIJNI, these predicates will fail if either the method fails (HRESULT != S_OK) or argument conversion fails.

Predicates are deterministic unless otherwise noted.

3.13.1                       java_create_object/2

java_create_object( +Class, -Object)

Given a UUID or Class string, creates an object and returns a reference functor.  If a Classname/UUID is given, it must be enclosed in curly braces, such as “{24345413-F98C-11D2-A93B-00C04F72E076}”.

If successful, the output value is a functor of the form

java_object(n)

where n is a unique integer maintained by the SWIJNI DLL.

3.13.2                       java_release_object/1

java_release_object( +Object )

Releases an object.

3.13.3                       java_invoke_object/4

java_invoke_object( +Object,
                  +InvocationAtomOrList,
                  +ArgumentList,
                  -Result)

Calls a method on an object.  The Object  must be a in the standard form and must contain a currently registered reference index.

The InvocationAtomOrList is either an atom (note, not a string or character list) containing the method name or a list containing the method name atom and, as required, a method invocation type atom.  See the section Types of Function Invocation for more information.

The ArgumentList is a Prolog list of arguments to be passed to the method.  These are converted as documented elsewhere.  If there are no arguments, pass the empty list (‘[]’).

The Result is usually an unbound variable, which is unified with the results of the method invocation.  If the function doesn’t return a value, it is unified with the empty list.

3.13.8                       java_release_all/0

Releases all stored references maintained by SWIJNI.

3.13.9                       java_context/0  (non-deterministic)

Sets up a “once”-style backtracking predicate that discards all object references upon backtracking.

In other words, all Java object references obtained during deeper searches are marked with a context “horizon”.   This predicate always fails upon backtracking, and, along with failing, discards any Java object references accumulated during goal searching descent.

3.13.10                  java_context_global/2

java_context_global( +Object,
                     ?BooleanAtomOrUnboundVar )

Promotes a local (context-specific) reference to a global reference.

If the Boolean (second) term is bound to the atom true, the object reference represented by the given Java functor is promoted to the global context.  Hence, it will never automatically be released due to backtracking.

If the Boolean term is bound to the atom false, the object reference represented by the given Java functor is marked as belonging to the local context.  Hence, it will be released when the java_context predicate that created the current context is backtracked over.

If the Boolean term is unbound, it is unified with true if the Java object is in the global context or false if it is not.

3.13.11                  java_list_to_date/2

java_list_to_date( ?DateAsIntegerList, ?DateAsFloat )

This predicate performs bi-directional conversion between standard OLE dates (such as used by ADO) and lists of integers.  For example,

java_list_to_date([1998,11,2,0,0,0],36101.0).

In this case, both terms are instantiated, so only a unification check is performed.  If the list term is a variable, the date term is converted to an integer list and unified with it.   If the date term is a variable, the list term is converted to an OLE date and unified with it.

The order of terms in the list is [year,month,day,hour,minute,second].

3.13.12                  java_errors_as_exception/1

java_errors_as_exception( ?Bool )

This predicate controls whether SWIJNI reports an reflect interface error as an exception (using the java_error functor) or simply fails the predicate. The default is to fail the predicate.

If the Bool term is unbound, it will be bound to the current setting of the treat-errors-as-exceptions flag, either true or false.

If the Bool term is bound to either true or false, this value will be applied to the internal flag.

Note that this is a persistent “side effect”, and is not undone during backtracking. 

This reporting mechanism currently only applies to java_invoke_object; that is, calls to reflect::Invoke (via DispInvoke).

It is recommended that setting java_errors_as_exception(true) should only be done during testing to track down the source of an interface failure.  Since many Java methods routinely report failure during normal operations (such as end of file or non-present properties), throwing exceptions during such benign operations can be seriously misleading.
