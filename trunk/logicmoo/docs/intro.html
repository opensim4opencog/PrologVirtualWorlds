Use a KIF/CycL like language  to represent shared activities among autonomic agents.  Java libraries integrate with a first person MOO system JaMUD to provide debugging and agent programming facilities. 

Symbiotic Refection between an Object-Oriented and a Logic Programming Language

Some purposes of LogicMOO:

1) Is to create a SHRDLU type environment for Modal Logic testing. 
    A way to put our knowledge engineering to a real virtual world test!

2) Provide a interface for humans to involve themselves in the lives of logically created agent creatures.

3) To represent episodic models of memory, diagrams for belief engines.

4) Provide an adaptable interface for a fire hose of data to be stored. And later wrapped into a single pragma for inference.

After OpenCyc has been released we are very excited to use it.  We had created a generalized API between Jamud and an "LogicServer" usually that entails first pragmatically pushing the Java system data into the LogicServer (in Prolog).  Then notifying updates via a wrapper around java objects. Using OpenCyc however opens up more possibilities:  Not just pushing data from Java to Cyc but "querying" for instances of Java member data. 

For those prolog hackers out there that want to write Prolog Agents like used in OOA projects. We are interested in using the HornClause libraries in Cyc. Maybe can get the best of both worlds and still make it feel like prolog.  Perhaps a Prolog to CycL HornClause tool?  Currently LogicMOO uses the Motel system of propositional Logic.  And other systems are being experimented with.

We don't plan on immediately using NL features of Cyc for game control but definitely will be fun using .say "Hello CycHumansMt, is Joe a Person?" type things since that seems to be the direction the Cyc is going at first. The Jamud command parser exports a very tight infocomm Zork like domain of imperative commands currently. It will be interesting still, mapping them to Cyc.. We would like to be able to eventually ask the 'BejeweledLady' to go and rent a room in the Inn but don't pay more then 50$ per night and have the agent run of and do the tasks based on a Cyc planning library.  We would like to see LogicMOO eventually accompany as a Cyc Example Application.

Java Libraries are provided that you need to use KIF (Knowledge Interchange Format) and soon enough DAML (DARPA Agent Markup Language) content into an episodic simulation inside a virtual world MOO.  


A simple example of agent orientated programming inside a Moo:

  • A Rain Agent that enters a room object and emotes that it is raining.  It then begins adding a 'wetness counter' property to each object in the room.  The longer it stays in the room the more it increments. (This does not have any functional impact on the current workings of the objects in the room)

  • Later you may want to write another agent who enters the room called 'Sun Agent' who's duty is to look for objects that have the property 'wetness' to decrement.  

  • A third agent called 'Common Cold' can go thru and look for both a 'health property' and a 'wetness property' and then decide to alter the 'smile method' from emoting 'beaming smile' to 'tries to smile from behind a stuffed up nose'.  

Why is this important?

As since all of these actions are moving thru representation (it can be depicted in triples or graphically) an agent may analyze the structure and build a 'story script'. The properties of such a script are in an ontological in nature and are best and are understood in a propositional language the represents an instantiated scene.  Since information about the methods and properties are public at the time the system may at later replay the scenario re placing objects in original scene in a non-deterministic way. 


As we discover new facts incrementally we learn the situation was not correctly represented.  Since it is persistable we can re-instantiate the episode and put better-educated agents to the task of understanding. 

 

Persisted Data Model

Cyc KB?  KIF File?  Daml Page?  Jamud running mud objects?

 

JaMUD is saved an loaded from an XML pages.  This means at any time we can serialize the  'MOOState' context (Microtheory).

This is meant to be a solid context comprising of instances and sets.  The other ontologies about understanding and using these can stored as well as hypothetical rooms and objects.  They in could be called 'Exemplars' for example the Unicorn Pegasus is a mythological creature and could live cut off in in the 'Giant Room Of Mythological Creatures' and looks and save like an instance of Unicorn.  But Pegasus' room may not be part of the relevant KB and therefore not effecting to the rest of the system.  But how would we persist information that does not fit into a 4D space?    For example these two rules: 

 

(implies 

    (and 

        (isa ?PERSON Human) 

        (acquaintedWith ?OTHER ?PERSON))

    (acquaintances ?PERSON ?OTHER SimpleContactAcquaintance)) 

 

(implies

    (spouse ?X ?Y) 

    (friends ?X ?Y))

 

Each rule is kept in an instance of an Assertion.  Physical Regions of the MOO may be effected by any 'Set' of instances from the Assertion Class.  Assertions can be divided into Microtheories that can be believed by agents (or Mobs).  The MOO reserves no special entity for enforcement of rules unless an agent has been assigned specifically to a room to do so.  Most cases agents in the room are picking from there own private array of Assertions which is separate from  an entity using the room's set.

KIF
We, like others, are comfortable with the good knowledge engineering coming from the standard upper level ontology working group.  And between the content produced there and Cyc should be able to come up with interchangeable mappings without conflicts between 'Entity' vs. 'Thing'. 

 

CycL

Asserting CycL directly like  "(isa Fluffy Mammal)" or an entire CycL Microtheory from File which becomes a set of very useful 'Assertions'

 

DAML

The inclusion of Daml ontologies (from web sources) to become available to search and inference.  

The user  may use existing tools for editing content before uploading to server.

Since JaMUD is saved an loaded from an XML pages it will be possible to make sure these are RDF/DAML compliant. 

Additionally RDFQuery should be useful.

 

SQL Emulation

Minimalist (at first) SQL Interfaces.  Mainly a SQL-XMLQuery and a system for inserting.  

Since it is possible to make the stored content look and feel like SQL Tables.  Perhaps leveraging inference.
