1.    What is a Virtual World?

Virtual Worlds are creative new contexts for multi-user interaction and collaboration.  Virtual Worlds is a client server application. Virtual Worlds lets users communicate with one another and interact with up to 30* others in real time.   Users can interact with and explore rich, multimedia environments (in a text, 2-D or 3-D graphical environment).  All objects can be scripted to interact with other objects in the world.  In addition, web integration lets world builders customize the client application using DHTML and put the application in a web page. 

 

Virtual Worlds supports the following:

1.  Multi-user, synchronous and asynchronous interactions

2.  Multimedia environments and users (both 2D and 3D)

Users can move in a 3D environment, navigating the space with mouse or keyboard.

        Supports Direct 3D rendering engine

        Autodeskâ 3D Studioâ[i] (.3ds) and  .X files and animations for environments

Supports any audio formats compatible with Internet Explorer, as well as .WAV and 3D audio via direct sound.

        Custom avatar creation kit (facilitates layout and formatting of graphical image)

                        2D sprited avatars--users can choose from a limited set of avatars

                        Avatar gestures--provides a limited number of explicit and idling gestures

3.  Persistent object store

        Able to remember changes to the world made in the past.

4.  Web integration

        Ability to customize user interface using DHTML and active scripting languages

        Supports Microsoft Active X technology

Fully scriptable objects, rooms, and avatars

 

Virtual Worlds limitations:

1.  Release from Microsoft Research as a prototyping platform for research purposes

2.  Supports Windows 95, Windows 98, and Windows 2000 users using IE5.0

3.  Scalability--limited to approx 30 users per room (simultaneously viewing 30 users when using graphical avatars)

4. 3D environments-- The 3D Virtual Worlds environment is being rendered in real time and is not pre-rendered from a particular perspective.   For performance reasons, we suggest you limit your design to about 2000 polygons (triangles) and 2 megs. of textures, sounds, and models per room

 

About this Document

This document is written for designers and beginning world builders.  In order to design a world, you need an overview of the basic concepts of the Virtual Worlds platform and you need to understand how the basic concepts that will effect how you design and build your world. This document will outline of the type of decisions you need to make about your design process and will help you estimate the scope of design work.  

 

You can use the Virtual Worlds platform to build multi-user applications that are text based, 2D or 3D.  This document focuses primarily on the design of content that will appear in a 3D world.  Here you will find specific information on how to design textures and models for your 3D world.  This document does not cover how to customize the user interface (design a custom application) of your project.  It also does not cover how to script and add behaviors to your world and objects.  Refer to the white paper on custom UI and the script examples for information on these topics.   Read the Virtual Worlds help documents for more help on specific features of the client user interface and programming API references.

 

This document is roughly divided into 4 parts. 

Part 1:  Virtual Worlds Building Basics.  Sections 1-3 give you a general overview of the basic Virtual World principles and design processes.  These are essential to understand before beginning to build your world!

Part 2:  Using the Wizards and World Kits.  Sections 4-5 give you an overview of using the wizards and world kits to customize your world.  For these first 2 parts, we assume no scripting knowledge.

Part 3:  Building Custom Worlds.  Sections 6-17, give you specifics on how to build custom worlds, specifically for designers building 3D graphical worlds.  Before building a custom world, first experiment by using the wizards and world kits. 

Part 4:  Appendix.  The appendixes  provide detailed information on specific processes like editing the Avatar files and tips and tricks when using specific modelers like 3D Studio Maxâ.

 

2.    V-World Building Basics

Design Process Overview

When designing a Virtual Worlds project, first read this document!  Then try building a small world or part of a world, customizing it, posting it, and letting someone explore your world. This will help you better understand the amount of work involved.  You can expand the environment later, but it's best to start simple, with a small modular part of your design.  Remember, building your world is only the first step.  Managing and maintaining your world is a big part of the work -- your audience will expect your world to be up and running, and be updated!  Below are some basic steps to building and deploying a world:

–                     Initial design work. 

Initial design work should take place without a live audience.  This means as you design the user interface, build the 3Dworld, add custom textures and models to your 3D world, make some custom avatars, write scripts to add behaviors to your world, etc. there will not likely be end users in the space, making changes or being affected by your changes as you are trying to develop the world.  See below for more info on the initial design work.

–                     Posting content on web servers and setting up your server.

In order for others to see your content, it will need to be accessible to them on a web server. While doing your initial design work, you may either choose to work running the Virtual Worlds client and server on your local machine or on a different machine.  You may choose to have all of your content (images, 3D models, sounds, etc) on the client or on a web server. 

 

Graphics stored on the client means, at some point, before you launch your world, the people who access your world need to download and/or install the content to their local machines.  Like most web pages, we assume your content (images, models, sounds, etc) will be served from a web server and the Virtual Worlds server will be accessible to those accessing your content.  Virtual Worlds does not automatically post your content to a shared server.  You will need to do that yourself using standard web posting tools.  Another option is to pre-install all the content on each client.

Note:  For performance reasons, we assume you are running the Virtual Worlds client and Virtual Worlds server on separate machines. 

 

During the initial design phase, if you plan running the client, server, and content on your local machine (this is the easiest way to get started) during the initial design phase, make sure to organize the content on your local machine so the content all "lives" in a few directories.  Best if you put custom textures, sounds, etc. in one set of directories that is easy to post, and that any paths to content files that point to local content use relative paths.  In other words, rather than specifying a texture lives in:

 c://program files\microsoft virtual worlds\localcontent\sdk\library\textures\color1.gif,specify the relative path:  \sdk\library\textures\color1.gif sothat when you copy the entire directory to post on a web server you won't have to go back and redirect all of your paths.

–                     Making changes once your world is "live".

We assume that once you post your world, as people are using the world, the state of the world will change.  For example, avatars may pick up objects and move them.  When they return to the space, they expect the object to be where they left them (unless someone else moved it), or your world might remember who's been there before and greet new users differently than frequent users, etc. 

 

Once you "launch" your world, it is "live" and is constantly being updated.  This means that at some point the script or otherwise "saved" version of your world that you may used to initially deploy your world will be outdated.   Regardless of how you initially designed your world (using a script or using the in world tools provided), you will probably need to modify your world from the "live" version if you do not want to loose many of the user generated changes in your world.   You will also want to make regular backups of your world at non-peak hours, from the server application.

Initial Design Work

We've found that world builders tend to use one of the following processes for their initial design work.  You will want to choose the right process depending on the scope of your project and the skill set of your team.

 

–                     Option1:  Build everything in world using the wizards, sample content, and authoring tools.

We recommend using the wizards and building in world if this is your first time trying out Virtual Worlds, if you want to get a world up and running quickly, or if you want to design and deploy a world that does not require a lot of scripting that is similar in design to the basic client. 

We also recommend modifying your world in world if you want the revisions to be immediately remoted to connected users without taking down the server, and if you want to update your world using the most current live version of your world.

It is recommended that you use this process to get started if you do not have a scripter or web developer on your team. 

 

–                     Option 2:  Build the world from script

We recommend building your initial world from script if you have already built a simple world using the kits provided, and are ready to build a complex world with a lot of scripting.  If you are using a lot of scripting in your world, we suggest you do your initial development work in a separate script, and build your own kit and integrate this into the Build a World Wizard.  We also recommend building via script if you have many different people working on your world.  This way individual people can "check out" different files and work on them separately.   Note:  Once you launch your world, you will want to update it using the inworld tools.

 

If you are using the inworld tools, there is a "save to script" feature in the Virtual Worlds Server application.  This feature generates a script for you for the world that you built using the wizards, and inworld tools described above.  This creates a "machine" generated script that is harder to read than a script you create by hand, which may be an issue if you want to hand edit this script.  If you plan on iterating many times on a script, and you are primarily as scripter, we don't recommend that you initially build the world in world--but that instead you use the script examples and modify those to build your world.  See the white paper on Custom UI and the programmers guide for more information on this.

 

A disadvantage of building by script means you aren't taking full advantage of the tools we've created to quickly create and modify objects.  For example, using in world geometry tools, you can easily move and place objects without worrying about their position in world, and save their information by saving your world.   If you build a world by script, you will need to enter the location of an object in its x,y,z position in the script.  If you choose to build a world via script, it is very likely that you will still use the in world tools and wizards to test things before copying the positions and locations, etc. into your script.

 

3.    Basic Design Principles and Guidelines

Overview:  The World, Room, Artifacts and the Object Hierarchy

Before beginning designing your own custom world, there are a few design principles and guidelines that you need to understand in order to design for the virtual world platform.  These are briefly described below. 

–                     World contains rooms, rooms contain objects

Each world must live on a Virtual Worlds server, and should contain at least one room.  All objects, including avatars (users), portals (connect rooms) must "live" inside a room. Basically, every item, including the world itself, is an object.  You can program the objects, including the avatars, rooms, artifacts, etc. using Microsoft Visual Basic Script Code or any active scripting language.  All the information on the objects is stored in the Virtual Worlds database, and changes are broadcast to all people running a version of the world.  The objects in a V-World have an object hierarchy.  The easiest way to view the object hierarchy in your world is to open the Object Explorer.  Access the Object Explorer from the author tab of the Authoring Client or from the properties page of an object.

 

 

 

 

 

 

 

 

 

 

 

 

 

–                     Rooms as conversation and rendering units

When laying out your world, think about what the users will do in the various rooms.  Rooms define the conversation area and rendering unit--they do not need to be spatially organized or literal in design. 

 

When designing your world, consider the layout of the rooms.  For performance reasons, we have designed the Virtual Worlds hierarchy so that everything in a room is "aware" of everything else in the room.  Typically, things are not aware of things not in the current room.  For example,in the example above if my avatar is in the Lobby Room in world MyWorld, I can see other avatars in the Lobby and all of the objects in the Lobby.  I do not see or know what avatars are in the other rooms (like the Conference Room above) in the same world.  For the rendering of the graphical world, only the objects in my room are rendered.  I do not render objects in other rooms until I enter that room.  This means as I move from room to room, I expect a delay as the graphics for my new room download (if the graphics are not pre-downloaded) and render.

 

–                     Suggested design guidelines (includes avatars, sounds, and objects inside a room)

As you design the layout of the rooms, think the room as a conversation unit (holds up to 20* standard graphical users, I only see and hear those in my room), and a rendering space (I only render objects in my current room), there will be a delay as I move to another room and the graphics are loaded.  If you are adding to the kit spaces (provide in the Build a World Wizard), or you are making a custom space, we suggest designing your 3D models with the following limits per room:  <2500 triangles and <2 megs of graphics and sound.  You need to be especially aware of download times if the content is not distributed on CD or otherwise pre-downloaded. 

 

Typically the graphics of the room are downloaded and rendered on the user's local machine, while the live conversation, user's position, and updates to objects in the room are all sent through the server and remoted to the connected clients.  The users hardware will effect the rendering of the space, and the user's network connection will affect the speed of the users communication, and updating spped.  See the Virtual World help for minimum hardware and software configurations.

 

Note:  All worlds must "live" on a Virtual Worlds server.   A server can hold multiple worlds.  In total, a server can hold up to about 30 users* at one time.  A server can hold multiple worlds, and the total number of users in all worlds on one machine should not exceed 30*.

 

* We are currently working on performance improvements.

 

Basic Programming  Concepts

There are a few basic programming concepts you must understand as you design and develop your space.  For more information on these concepts, refer to the Programmer's Guide in the help docs.

–                     Object Oriented

AS described earlier, the basic objects include avatars (users), artifacts, and rooms --and they all must exist inside a world.  Every object in Virtual Worlds has a set of properties and methods that defines its state and behaviors.  In addition, every object in Virtual Worlds can inherit properties and methods from its exemplar (base class).

   Objects must exist in a world in a specific location.  In the diagram of world "MyWorld" shown above, there are two rooms.  Each room has several objects such as a portal to another room, avatar, and various artifacts such as the desk.  The avatar also contains several objects including the hat.

–                     Scriptable

To allow for flexibility, every Virtual World object has many properties and methods.  You can add properties dynamically, and affect almost everything in the world via script.  Scripts should be written in VBScript , JScript or other active scripting language, and can be composed as text files in the editor of your choice. 

 

To help users learn to script we have provided various scripting examples and documentation.  One quick and easy way to learn scripting is to review the sample scripts, and to walk through the script tutorial kit in the build a world wizard.

–                     Client-Server Architecture

Information about an object is held both on the local machine and on the server machine.    The client and the server can be run on the same machine, but we assume at some point users will be connecting remotely from their local machine to the server.  Properties and methods can be changed and executed in either or both places, at the programmer’s discretion.  For example, the 3D rendering in the view window is done on the local client machine, while the position of the avatar is sent to the server and remoted to the other users currently in your room.  For performance reasons, it is important to understand when to put information about an object on the server and when to put on the local machine. 

 

4.    Using the Wizards and World Kits: Getting Started

Build a World Wizard: Creating, Saving, & Deleting Worlds

To get started, best if you get a simple world up an running.   The easiest way to begin if you do not already have a world up an running is to create a new world using the Build a World Wizard and run both the Virtual Worlds client and the server on your local machine.   Running locally, you can go through most of the topics described in this paper.  For more help on this see the "Create a World Wizard" help and the "Virtual Worlds Server Interface" in the" Virtual Worlds User Interface Reference." 

 

Quick overview for creating, saving and deleting worlds:

 

1.  Run the Virtual Worlds Server application.  After installing Virtual Worlds, launch the Microsoft Virtual Worlds Server from the Start menu.

2.  Run the New World Wizard.  Launch the new world wizard by selecting "New World…" from the File menu.  Enter "localhost" for the server location on page 1 of the wizard and in the content server field on page 3 of the wizard.  If you do not have a web server set up, leave the server address on page 2 blank.  Note:  You cannot run locally (localhost) if you do not have the server running on your local machine.

3.  Create the world.  Click "finish" on the last page of the wizard.  This creates the world with objects and design you selected, launches the Authoring Client, and puts you (your avatar, the world owner) in the entry room. 

4.  Go to the authoring tab of the client.  Try using the tools, creating objects, modifying objects, etc.  Note:  The Standard Client does not include the authoring tab.  We assume you may not want your end users to have access to the authoring tools, so we created a separate client that does not have these tools.

5.  Your world is automatically saved as you make changes to the world.  To explicitly save a version of your world, go to the file menu in the server application and select save (to backup your world) or save as (to save your world to script.)  All of the worlds you have available on your server appear at the bottom of the server application window.

6.  To re-enter a world you have created, run the server application, open the world, and log on to that world using the basic client user interface.  Note:  for destination, enter localhost/<worldname>.

7.  Invite someone else to your world.  The easiest person to invite to your world is yourself!  Try launching another copy of the client on your machine.  Create a new avatar (avatarname) and enter your world: locahost/your worldname.  Move around in the client using your arrow keys and you should see your other self in the world.  Since you are running 2 versions of the client and the server on your local machine, the performance may be very slow!  Close one of the clients when you are finished to improve performance.

 (Note:  Others can enter your world if your machine with the Virtual World server is available to them, they have the Virtual Worlds software, they are running on a PC with Internet Explorer v5, the world graphics are available to them, and they know the name of your world)

 7.  If you want to make your world publicly available, display your world in the Virtual Worlds directory by going to the tools menu in the server application and select "Show World in Directory Services."  By default, your world does not appear in the Directory.  Note:  Directory services require additional socftware.  Please contact vworlds@microsoft.com for more information.

8.  To delete your world, select it in the server application (if you have created multiple worlds it may appear minimized at the bottom of the server app) and select Delete from the file menu in the server.

 

Congratulations!  You have gone through all the basic steps in building a Virtual Worlds!

Room Wizard:  Creating a New Room

To create a new room, launch the room wizard from the authoring tab of the Authoring Client.  The room wizard creates the room as well as objects inside the rooms like the portals and various artifacts such as the floors, walls, etc.  See the help docs for more info on the room wizard.  If you built a world using the world wizard, the wizard automatically creates a world with at least one room.  You see and can talk with others that are currently logged into your world and are in the same room.  Remember, you will need to create a portal and set a destination for the portal if you want users to be able to go between the rooms.

Object Wizard:  Creating a New Object 

To add a new object, launch the object wizard from the authoring tab of the Authoring Client.  Once you pick the type of object you want to create, the object wizard creates the object for you, places it in front of you in the 3D view, and opens the property pages for that object.  Using the property pages, you can edit the name, description, geometry, texture, and etc. of the object.  When you edit a property, all users connected to your room see these changes live.  For the full list of properties and methods of that object, open the object explorer (advanced button).  For more help on the Object Wizard, see the help documentation.

 

–                     Object types

The set of object types listed below can be created using the Virtual Worlds Object Wizard.  Different object types have different properties and methods.  Note:  You can create your own custom object types.  We also provide other object types that have not been added to the object wizards (for  example, the light object).  To create you own object type or to create a light object, see the Virtual Worlds help docs.

 

Artifact

The artifact is the generic object type.  Most of the objects in the worlds you generate using the Build a World Wizard are of the type artifact.

        

Interworld Portal

There are 2 types of portals, inter world portals (between worlds) and intra world portals (between rooms within one world).  When creating Interworld portals you can specify the destination server and world in the portal property page.  The world you enter will use its default entry room and entry point to position the avatar in that world.

 

When going between worlds, the avatar will not be able to carry its objects (if it has any).  The entry world may also limit the entry of avatars that have certain costumes, names, etc.  While the user connects to the server, logs in (if needed) and loads the content for the entry room, a transition (spinning cube) will play on the client machine.

 

Room Portal

When creating a room portal, you can specify the destination room (in your world), and the entry point and facing direction.   While the user loads the content for the new room a transition (spinning cube) will play on the client machine.

 

State Machine

When creating a state machine, you are creating an object that can have multiple states.  You will need to use scripting to set the various states.

 

Web Page

When creating a web page object, use the web page property page to specify an url to connect to.   Clicking on the "show…" menu item will launch the url in a separate window.

 

–                     Kit spaces:  Editing objects in the rooms tab that have been set to IsUIVisible

In order to simplify the kit worlds, we "hid" the names of many of objects in the rooms that an end user would most likely not interact with.   This means users cannot right click on the object to view its properties and the name of the object is not visible in the room’s tab of the client user interface. You can "unhide" the names of these objects to edit them. 

 

We built the spaces out of many different objects (parts) to make the kit as flexible as possible for world designers.  For example, you can edit the individual walls, backgrounds, and floor colors of every space you create using the wizards! To view the property pages of these objects, you will need to make their names visible in the user interface (IsUIVisible property has been set to false, you need to set to true).

 

If you would like to edit the walls, floors, or other objects in a room using the property pages (change textures, scale, etc.) you will need to set the IsUIVisible property of the object you want to change to true.  To do this, open the Object Explorer from the author tab of the Authoring Client.  Find the object you would like to change.  Look in the properties for IsUIVisible and change this property to true.  The object should appear in the rooms tab and when you right click on the object, its menu will appear.  From the Object Explorer you can change all the properties and methods of the object.  You can also select properties to open the properties dialog.

 

Using the Object's Property Pages

You can quickly modify the look of your objects by changing the geometry and texture properties using the object's Property Pages.   The pages below describe property pages that are standard for all object types.  As noted in the previous sections, each object type may have additional property pages.  For more basic help on the property pages see the Virtual Worlds help documentation.

–                     Using the Texture Property Page

To apply a texture to a geometry in Virtual Worlds, open the object's property page by right clicking on the object and selecting "Property…"  Then select the texture page. 

All surfaces (meshes) of an object can be textured.  In the texture property page, select a surface to texture using the property pages (by default, all surfaces are selected). 

We have provided a limited material library.  All of the textures that appear in the kits appear in the directories _kitname.  We also have provided a set of gradients, grayscales, naturals (included an animating water and fire texture), and a transparent texture.  You can view any of these textures in the property page and distribute them with your virtual world.

You can color any mesh any color by typing in the RGB value of the color.  This will blend the color with the texture.  If you have a dark color, often the texture will not be visible when displayed in Virtual Worlds. 

 

If you want to use your own custom image, you can dynamically map any .gif (including animated .gif), .bmp or .jpg file onto any surface (mesh) of any object in your world by typing in the path or the url into the text box.  Note:   Files that exist only on your local machine will not be remoted to the other viewers.  You will need to post your local images to a web server in order for others to view those images.  For more help on how to build custom textures, see the custom texture sections of this document.

 

–                     Changing Geometry using the Geometry Property Page

To change the geometry (model assigned to an object) in Virtual Worlds, open the object's property page by right clicking on the object and selecting "Property…" ; then select the geometry page.

We have provided a limited model library.  All of the models that appear in the library are either models from the kits spaces, models from the Microsoft DirectX model library (from Viewpoint) that have been simplified for use in Virtual Worlds, or generic objects (walls, floors, primitives, lights, etc.) 

From the geometry page, you can also specify how you would like the object rendered.  Depending on your object and the lights in your room, different styles will have different effects.  The render styles include wireframe, solid (lit and unlit), gouraud, and phong.

 

You can add your custom geometry  (.X or .3DS format) by typing in the path or the url into the text box.   Note:  Files that exist only on your local machine will not be remoted to the other viewers.  You will need to post your local images to a web server in order for others to view those images.   For more help on how to build custom geometries, see the custom model sections of this document. 

 

–                     Approach Property Page

Rather than navigating you avatar to a specific location, you can right click on an object and select "Approach" to automatically navigate you to the object.  This will bring you a fixed distance (by default it is 4 units) from the origin of the object.  If you have a large object, like a house, this may "land" you in the middle of the object or up in the air--rather than in front of the house.  To adjust the approach distance, use the Approach Property Page.  When you set the approach distance, make sure that the position where you "land" the user does not fall outside the collision detection boundary.  If this happens, the user will not be able to freely navigate back inside the space.  If this happens to you while designing, "approach" another object to "escape."

Note:  At this time you are not able to adjust the Y axis approach distance in Virtual Worlds.  The origin point can be set in the .X file using your modeling program to influence approach on the Y axis in Virtual Worlds

 

In addition, when you approach an object, you can specify a "random scatter distance."  The scatter distance randomly places the avatar within <scatter distance> from the object to prevent avatars from landing "on top of one another" as they approach an object.

 

Finally, you can specify whether or not to approach an object from the front.  Some objects (like sprites), do not have a "front" or "back"--but instead always turn to face the user.

 

–                     Motion Property Page

Using Virtual Worlds, you can easily create a spinning object by checking the object spins checkbox on the object's Motion Property Page.  The spinning objects spin around the center of the object on the axis (or rotation) you specify.   Grouped objects will spin around the center of the group of objects.  Using this page, the rotation rate can be used to change the speed.  To spin in the opposite direction, use negative numbers.  More complex animated behaviors should use the StartAnimation method.  For more help on this, see the Virtual Worlds help docs.

5.    Avatar Overview

The term avatar, often used in Science fiction literature, is used in V-World to describe the representation of the user in the virtual environment. 

 

We have provided several types of "special" avatars.  To help run and manage your world, we have designed avatars with special privileges.  These avatars include wizards, owners, hosts and authors.

In most ways, the avatar is like other standard objects.  Unlike other objects, the avatar object can move between worlds.  Although objects the avatar is carrying will not be allowed to move between worlds, and specific worlds may restrict entry of avatars, it is possible for the avatar object to migrate between different worlds on different servers.

Avatars with Special Privileges--Wizards, Owners, Hosts, Authors,

When running a world, we assume that these will be different types of people (avatars) involved--these functions include wizards, owners, authors, and hosts.  Wizards can affect everything in a world.  They can change all objects.  Owners own specific objects in a world (the world owner owns the world)--so for example, I can create a new object and I would be the owner of that object although I do not own the world.  Authors have the ability to view the world using the authoring client (the author tab is visible)--but no special privileges.  They cannot edit any object they do not own.  Hosts manage online behavior and are able to kick users that aren't behaving properly. 

 

When designing avatars and roles in your world, keep in mind the different types of user roles and how you might want to visualize (or not visualize these different roles.)  For more information, see the Virtual Worlds help docs. 

 

Avatar Object Overview

Like any other object, all avatars can have graphical elements and informational elements.  With the Virtual Worlds platform, we provide a "standard" avatar.  To help world builders build avatars, we have provided the Avatar Construction Kit.  This tool will help you format your own custom images and gestures for an avatar using the default graphical representation of the avatar. 

 

The graphical elements of the avatar include the avatar image, gestures, and a point of view.  With the default avatar--these is a sprited graphical element.  As you move around the avatar will have a front, back and various side images.  The avatar image can be abstract or literal.  The avatar also has gestures (avatar can perform "actions" with graphical representations such as wave, smile, shrug, etc.), and point of view (camera view can be through my eyes, from a fixed point, behind my shoulder, etc.) 

 

The avatar should also have a thumbnail image.  This image appears in the member pane of the standard client.  If no thumbnail image is present, the default image is used to represent the avatar in the member pane of the standard client.  If you are using the Avatar Construction Kit, the thumbnail will be automatically created for you.

 

The avatar also consists of information-- including a text description and details, friends and ignored users, and a list of things (like the hat in the example above) that the avatar is "carrying."  For more information, see the avatar section in the help docs.

 

To make a custom avatar, follow the instructions in the Avatar Construction Kit.  Then put your custom avatar .gif, thumbnail image and .spr file available to users.  To test your new avatar graphic locally, place the three files in:  c:\\Program Files\Microsoft Virtual Worlds\Local Content\Avatar Graphics or create a new directory for your world:  c:\\Program Files/Microsoft Virtual Worlds\Local Content\Worlds\<World Name>\Avatar Graphics.

 

Avatar Graphics, Using the Avatar Construction Kit

 

The easiest way to make an avatar graphic is to create all the images for the avatar using a paint program like Paint or Photoshop, and assemble them together using the Virtual Worlds Avatar Construction Kit (ACK) application.  Avatars graphics are animated.gifs that consist of multiple frames (for example a front, side, back, wave, smile, etc.)    The ACK will let you lay out the avatar gestures and assemble them in an animated .gif, create a preview image for your avatar (the thumbnail file) and will create the .spr file (text file that describes the gestures, and gesture timings, and views).  Follow the instructions in the Avatar Construction Kit for help with the avatar tool.

 

The output size of each avatar image is 128 x 128 pixels per frame.   This does not mean that every avatar must fill frame--or that your original artwork imported into the ACK must be 128x128.

 

The transparency of each image should be set in your paint program.

 

There are up to 24 total frames per avatar.  The avatar gestures only on the front.  When viewed from behind, the frame “rear” is displayed, and side frames are displayed when viewed from the side to simulate a 3D character.

 

–                     Viewing your custom avatar in Virtual Worlds

To view your custom avatar graphic, put you avatar in the Virtual Worlds avatar directory: C:\ProgramFiles\Virtual Worlds\Worlds\<your world>\avatar graphics.  Launch Virtual Worlds from the Start Button (Programs\Virtual Worlds\your world) and select an avatar graphic after you log onto Virtual Worlds.  Select the name and graphic of your new avatar from the area associated with your world (not general avatars).

 

Note:  You will have to publish your files to a web server or share before others can see or use your avatar graphics.

 

To see yourself, change your camera view using the camera button at the top of the client user interface.  This way you will be able to view your avatar as it appears in the world as well as in your self view window.

 

Note:  For more on creating custom gestures and avatars see the Custom Avatar section of this document, and the programming guide in the help docs.

 

Common Questions when Designing Avatar Graphics

Below is a list of common questions and answers for designing avatar graphics.

 

What character designs work best for the avatar graphic?

When working on the character design of your avatar graphic, keep in mind, that unlike designing movie or cartoon characters, end users will be representing themselves by these images.  We have found that unattractive avatars are rarely selected and subtle gestures are often hard to see.  We highly recommend incorporating the design of the avatar into the design of the space.   We have found that people, who “inhabit” the avatars are the “life” of the space.  The more expressive avatars have been our users’ favorites.

Flipping between frames creates avatar gestures.  We recommend using expressive, exaggerated gestures; they are more visible and more popular.  We recommend testing avatars in V-World both to determine which avatars are “favorites” as well as to test image quality.

When designing the default frame, keep in mind that this frame is the most prevalent representation of the user.   It is advised not to have “strong mood” associated with the default frame.  For example, if the default is an extremely happy pose, it may appear odd when the user wants to display and angry gesture, as after showing the angry gesture, the avatar animation returns to the default image.

 

How many avatar graphics should I make and what will this cost?

The number of avatar graphics designed for a space will depend on the overall concept of the space.  Also the cost of the avatars graphics will depend on the design.  As a guideline, expect an avatar with 25 frames to cost between $750-$1500 per avatar graphic.  (This includes character design, animation, ink and paint, and cleanup.)    Note:  This figure does not include testing of your avatar graphics.

 

Can the avatar graphics that come with the Virtual Worlds software be re-distributed?

With Virtual Worlds we provide about 25 avatar graphics.  You are free to use and distribute (or delete) any of these in your worlds. 

 

Is it possible to design avatar graphics that only my actors/hosts, etc. can choose?

If a content provider does not want end users to select a specific avatar graphics file, using scripting you can prevent regular users from using specific avatar graphics. 

 

Can end users customize or create their own avatar graphics?

The avatar file format is a standard animated .GIF file.  Users are able to open an avatar graphics file and modify the file or create a new file.  In order to be seen by other connected members, the end user will need to post the avatar graphic so other connected viewers can see this avatar graphics.  We have found that end user avatar graphic creation is an extremely popular feature.  If you want to let end users create their own, you will need to build a mechanism in your world that shows end users where and how to post their avatar graphics so other users can view.  At this time, we are not supporting a formal end user avatar construction kit.

 Note:  When letting end users create custom avatar graphics, you may need to design a set of rules that limits the design of avatars.

 

Why does my avatar graphic has "fuzzy edges"?

When creating images for your avatar, make sure to "clean up" any edge of the image that is next to a transparent area.  This edge must not have an anti-aliased edge.  To check your avatar, check each frame of each avatar to make sure the edges of the avatars are not “fuzzy,” and to make sure the animations are correctly registered.

 

Can users change their name in Virtual Worlds environments?

End users can select their name when creating a new avatar using the login page.  The name of the avatar is a property of the avatar.  To prevent spoofing, users cannot change the name of the avatar.  The avatar graphic can be changed any time during the session using the avatar property page.

 

Can I change the look of the nametag that appears over the user's head in the 3D View?

The username appears as a nametag above the user’s avatar.  Wizards can dynamically change an avatar's nametag using the label properties to change the nametag using the Object Explorer.  You can also modify the nametag via script.

 

What is the difference between idling and explicit gesturing?

Virtual Worlds avatars will idle, varying slightly, when they are “paused.”   These idling behaviors cannot be controlled by the end user and will vary depending on the content provider’s design of the character.  Each character will have several idling behaviors.   For example, these behaviors can include looking left/right, blinking, scratching head, nodding head, etc.

End users can communicate to one another by explicitly controlling their avatar’s gesture.  Using the Avatar  Construction Kit and the basic client, the Virtual Worlds all avatars support the following gestures:  Greeting; Happy; Flirt; Laugh, Shrug; Frown; Flirt; Sleep, Action, Pose, and Turn Around.

 

Can the avatars remain in a “mad” state, or a “happy” state?

We have not designed the standard avatar to support persistent states.  The only "sticky gesture" is sleep.  Users fall asleep and stay asleep until they select another gesture or turn off the sleep gesture.  All other gestures return to the default state.  You can design persistent state avatars via scripting.

 

How do end users control their avatars’ gestures?

To control the gestures, the end user clicks on a button under the self-view in the client application. 

When the user clicks a button on the gesture toolbar, (or users a control key) the avatar “flips” to the associated frame and “plays” according to the associate time.  The ACK avatars have been designed to work with the Standard and Authoring Clients. If you use custom gestures or avatars, you may need to modify to client so the appropriate icons, tooltip, and gesture text appears in the user interface. 

 

Can I design 3D avatars or sprited avatars, or customize the gestures?

You can design 3D avatars or customize the gestures and the timing of gestures.  See the section on building your own custom avatar in this document for more information on the custom gestures.  Virtual Worlds supports 3D avatars, but this is not covered in this document.

 

It is possible for to create custom gestures with unique timings per avatar by modifying the .spr file. (Note: see section on custom avatars in this doc for more help).  Nevertheless, when the user clicks on a button in the gesture toolbar, the tooltip and icon will not reflect the preset gesture, and the chat history pane will display preset gesture.  Depending on the design of the space this may be a desired effect.

6.    Creating Custom Worlds

In many ways the design process for Virtual Worlds environments is more like set design than architecture. When designing an environment, think of the avatars in the scene, and the types of activities and conversations that may take place in the scene.  

 

For many reasons, we recommend creating simple environments.   Due to the real time rendering of the scene, performance is a critical issue. Performance of rendering is greatly affected by large texture maps and complex geometries.  Therefore designing simple geometries and small textures is absolutely crucial to the design of a successful Virtual Worlds environment.  Carefully construct your models and textures, and check performance often by testing in Virtual Worlds.  There are many methods for simplifying spaces.  For example, incrementally lower the resolution of your textures and test in Virtual Worlds to determine the lowest acceptable resolution of individual textures can assess image quality of texture maps vs. performance in Virtual Worlds.

 

As Virtual Worlds are distributed by downloading over the Internet, the overall size of the Virtual Worlds (environments, avatars, and sounds) should be limited. Remember, the total download time of your space should be an absolute maximum of 2 megs compressed.  This includes all environments, the textures, sounds, and all avatars.

 

When running over the Internet, up to 20 different avatars can occupy a room. We highly recommend incorporating the design of the avatar into the design of the space, and testing both your avatars and the general avatars in your environments.  Make sure that users can follow the conversation when your maximum number of people is in the room.  Check to make sure the visual complexity of textures and geometries in the environment do not overpower the avatar design, and check that the scale of your environment is not too small or large in relation to the scale of the avatars. 

 

Finally, for the first custom Virtual Worlds environment you create, we highly recommend creating a simple space to better understand the production process of Virtual Worlds.  Although creating each element may not be difficult, making sure all of the parts work together to make a cohesive, compelling environment is challenging.

7.    Creating custom Rooms and Objects

Building Custom Rooms and Objects

When designing rooms and the objects inside rooms, like designing the room layout for the world, remember that the room is a conversation and rendering unit--users see and hear everything in your room, and are not typically aware of objects (including avatars) outside the current room.

 

When building the geometry for custom rooms and objects in your 3D modeling package, it is very important to first determine when to build the space as one model and when you should to break up the 3D model into separate pieces.  In general, it is most convenient for the 3D modeler to build a custom room as one model, but it is most flexible to control different objects via script by breaking the model into separate individual pieces.  If you want to control a specific object (for example, if you want someone to be able to pick up a brochure and carry it to another room), best to make a separate object with its own geometry.

 

For example, if I wanted to be able to move the chairs around a table in Virtual Worlds, the chairs and tables should be separate .X files, each with its origin point set in the center of the mesh, and each assigned as the geomtry property to a Virtual Worlds object.

 

I could either build the table and chairs in one scene and break it out into separate pieces (resetting the object origin of each piece), or I could build each piece separately.

 

Initially building the objects as one scene has several benefits.  You can more easily design and compose the scene to flesh out scene attributes.  This model can be considers a master file, which all designers can use as a common reference for lighting, textures, object position, etc. You can also have multiple people designing the scene, and another person doing the production work and breaking the scene apart to prepare the models for Virtual Worlds. 

 

To separate the master/template model into separate objects, select an object (including all of its meshes) inverse the selection, and delete the extraneous objects.  Save the file with a new filename.  Center and export the object from the new file.

 

–                     Designing a Room for Collision Detection

You can use the Boundary editor tool to create boundaries, and floor and ceiling heights in your room.  When designing your room, if you intend to add collision detection, make sure to design enough "clearance" so the avatars can easily navigate your room.  The collision detection typically needs to be placed x units in from the edge of the wall to prevent avatars from "breaking through" the walls.  Make sure to avoid building concave corners where an avatar is likely to get stuck.

 

Note:  If you are using the wizards and tools, to add a room to your existing world using the tools and wizards, it is easiest to create a simple or empty room using the room wizard, delete any unused objects if needed, and replace the geometry in the room with your custom model. 

Entry Point

When designing your room, decide where users will enter the entry room, and what direction they will face when entering.  If you have many users entering your room, make sure to set a scatter distance around the entry point.  This will randomly scatter users around the entry point to prevent avatars from "bunching up on top of one another."  If users are entering in a narrow space or you want to carefully control the entry sequence into your world, you will want to make sure the entry location is precise and adjust the scatter distance accordingly.  If no entry point is specified the entry point will be set to the origin (0,0,0).of the room.

8.    Creating Custom Textures

The colors in Virtual Worlds will not look exactly the same as they appeared in your modeling environment.  For this reason, it is highly recommended that you frequently test your environment and avatars in Virtual Worlds, and design for the Virtual Worlds environment.

 

It is suggested you keep copies of your textures in the highest bit depth possible.  As you test performance, and as CPU speed and bandwidth increases, you may find that you want to modify the bit depth of your textures.

 

Adding custom textures to the texture library

 

If you will be using the same local textures often, you may want to add you own images to the texture library (rather than typing in the path as described in the Texture Property Pages). 

 

You can open and modify any of the images in the texture library using any paint program like Paint or Adobe Photoshop.  If you installed Virtual Worlds in the default directory on your local machine, these files can be found in the following directory:  C://Program Files\Microsoft Virtual Worlds\Local Content\SDK\Library\Textures.  You can add your own textures by dropping a .gif or .jpg in any of these existing directories, or you can add a new directory and put your textures in your new directory.  The texture property page is hard coded to look in this directory for image files.  

 

Warning:  If you reinstall Virtual Worlds this will overwrite any of your custom image files.  Make sure to backup your custom image files when upgrading or reinstalling Virtual Worlds

 

Texture Size and Performance

When rendering the virtual world, to optimize the rendering performance and the download time, you will need to carefully design size of your textures.   Try experimenting by reducing the sizes of your textures and to see how small you can make your textures.  For the kit spaces and texture library we assumed people may be downloading these worlds, so most of the textures are 36k or below.  To reduce the loading time of the room, re-use textures when possible with the option of using mesh color   blended with texture for variation.  Remember, we recommend no more than 2 Megs for all content in a room (including avatar graphics, textures, and sounds), and make sure to test the performance of your space.

 

Because users move in the space, they will be viewing your textures from far away as well as close up.  Make sure you test the resolution and size of your texture from different viewing positions.

 

When you are creating textures, the most efficient textures are powers of 2 in size, for example 2x16, 8x32, etc.  A texture of any size can be texture mapped to any mesh, but if you are designing a very efficient space, keep this size tip in mind.

 

To prevent poor performance, we recommend using and image resolution of 72 dpi.

 

Designing Textures Viewed in "Motion"

–                     Avoid textures with thin line and high contrast

Textures with thin lines and/or high contrast will tend to "flicker and/or disappear" when they are viewed in motion (as an avatar moves in the space).  To avoid this effect, test your textures inworld and adjust accordingly.  For example, rather than having a thin black line next to a thin white line, try making the lines thicker, or try using a slight gradient between the black and white lines.

 

By default, for performance reasons, we render the world so it reduces the resolution of the screen when the user is moving.  When a user stops moving, the screen is rendered in "full resolution."  As you test your textures, make sure to test the rendering as you move around the space.

 

–                     Avoid textures which create visual noise patterns

Avoid texture that can create "noise" patterns to avoid moiré effects.  For example, if you have a texture that is a very small grid (for example, alternating pixels of color) as the renderer scales this texture while you move in the space, the texture can become distorted.  Like with thin lines and high contrast textures, make sure to test the texture in-world as you move around the space.

 

–                     Using motion to your benefit

We have had designers use the motion to their benefit.  If you want something to shimmer or look like a moiré while an avatar is moving, you can specifically design your texture to use these effects.  For example, a designer once designed a water texture with high contrast that seemed to "shimmer like water" as you moved.

 

Designing Tiled Textures

When designing a texture that tiles horizontally, vertically or both, use a paint program like Painter or Photoshop to help design tiles or make a large image and paste multiple copies of the texture to design the image.  These tools can help you design the image so it correctly wraps around itself.  The design of the object's mesh (es) will determine how the texture is tiled.

 

Note:  In Photoshop, you can use the offset filter by selecting "wrap" and painting in the seam(s).

Common Questions

Below are a list of common questions and answers  about creating custom textures.

 

What file formats are supported?

Worlds uses the following file formats:  .bmp. .jpg., .gif and animated .gifs.   Only .jpg and.gifs will preview in the texture property page. 

 

How do I create a texture with transparency?

To create images with transparency, you will need to convert your image to a .gif assign a transparent color using a tool like the GIF89 exporter that comes with Adobe Photoshop.  A common mistake is to anti-alias the edge of the image when creating transparent areas.  If you do this, you will see "halo" effect around the transparent area.  To avoid this, don't anti-alias the edge of the image.

Note: You can also avoid the jaggy edge by coloring the edge the same color as the background color of the scene.  To do this anti-alias against a known background color and then change the background color to transparent.

 

How do I create an animated texture?

To create an animated texture you must create an animated .gif by using any tool capable of doing so. Your animated .gif will continuously cycle through the animation a speed dependent on the user's machine.  This means that users with slow machines will see the animation play more slowly than someone with a fast machine.  You can set an animation speed of the animation using your .gif animation creation tool.  If an animation speed exists, Virtual Worlds will play the animation at the speed. 

To control your animation so it plays non-consecutive frames or so it plays x number of times then stops, etc., you will need to use scripting.  If you want detailled control of your .gif animations refer to the section in sprites (.spr files) in this document.

 

Note:  If the animation's frame per second (FPS) is more than the user's client can handle, the animation will run more slowly (at the system clock speed).

 

How do I create a texture with thousands or millions of colors?

Depending on the users screen setting Virtual Worlds is rendered in 256, high color (thousands), or true color (millions) of colors.   The .gif file format only supports 256 color images, and using. gifs, each image can have a different palette.

 

If you want to use an image with more color depth and no dithering, you can use the .jpg format.  This will let you display images such as gradients, etc. with more colors.  The .jpg format does not support transparency because of its compression.

 

9.    Building Custom Models

Building a 3D custom space that looks good and performs well when rendered in real time is a challenging design problem!  This section describes in detail how to build a custom model ready for Virtual Worlds using your 3D modeling program, and how to bring this custom model into Virtual Worlds.  Even if you are an experienced modeler, we strongly suggest you start with a very simple model and try brining that model into Virtual Worlds before beginning to build your own complex models.

 

This document will not teach you how to build 3D models.  We assume you are either already an experienced modeler or you will use the documentation included with your 3D modeling program to learn. 

 

Remember, an alternative to building your own custom models is using the kit models we've provided in the model library (see the Geometry property page) and assembling them together using the Geometry editor tool.

Setting the Origin of your Model

When building custom models, be sure to properly set the origin of the model.

 

When you add an object, the Virtual Worlds platform uses the origin of the model in a variety of ways.  For example, if you rotate an object, in Virtual Worlds, the object will rotate around the origin of the model.  If you drop and object, it will fall to the origin of the model on the Y axis.  If you approach an object, you will move towards the object and end your approach at the approach distance you've specified from the origin of the model. 

 

The origin (0,0,0) of the object is set in the model an object.  You cannot change the origin of a model from within Virtual Worlds.  Different rendering packages and different file formats often set the origin of an object in different ways.  For example, if while building you have moved the local coordinates of the object (or a piece of the object), it may not match the origin coordinates.  This may cause your model or parts of your model to import into Virtual Worlds off center or off screen.

 

We suggest you check and make sure you are correctly setting the origin point by building a simple 2 part object (like a house with a chimney)  into Virtual Worlds.  Note:  For 3D Studio Maxâ, see the Setting the Origin Point of an Object appendix for detailed instructions.

 

Note:  Depending on the object, you may want to set the origin of the object to the bottom (so it drops properly on the floor) or in the center of y axis of the object.

 

Note for 3D Studio Maxâ users:  Before you can export an object from 3D Studio Maxâ into Microsoft Virtual Worlds, reset the pivot point of the object, apply a material to the object, and export the object.  Resetting the pivot point insure that the pivot point matches the origin.  It does not insure that the local coordinates are centered on the origin. 

 

Scale of Object

Make sure to text the scale of your model in relationship to the scale of Virtual Worlds.  Typically an avatar is 2 Virtual Worlds units tall (one unit equals one meter).  This scale can be set and adjusted in your modeling program.  To check the scale of your model, make a box that is 2 units tall, import the box to Virtual Worlds, move your avatar graphic next to the box and view the world from a third person point of view using the camera view page or using the geometry viewing tools.

 

For most 3D modeling programs, make sure your grid resolution is set to <1.  This should work for most objects. 

 

You can scale objects in Virtual Worlds using the geometry tools, but it will save time if you build your models to the correct scale from the beginning.

Texture Mapping:  Meshes and UVW Mapping Overview

As mentioned in the custom texture section, the mesh you design and the material editor you use in your 3D modeler will determine how many textures are applied to the object and how the texture is projected onto a mesh.  Typically there is one texture for each mesh and one or more material (shading) per mesh.  For example, if my object is a wall, I could have one mesh with a texture stretched across the front, one mesh with another texture across the back, and one with a third texture around the edges.  For the front of the wall, instead of stretching on image across the front, I could have tiled the texture in the mesh in a variety of different ways (for example:  a series of panels, a small square grid, a large grid, etc.) 

 

In addition, you can design your model to use multiple meshes, each mesh having its own color.  This mesh could be colored to influence a common texture.  This will automatically build lighting and/or ambient color into the model.

 

If you are using existing models, you will not be able to change the name, number, or tile design of the meshes in Virtual Worlds.  You will need to do that in a 3D modeling software package.

 

When applying textures, if you have different people applying textures and creating the 3D models, it is essential they work together to design the models to achieve the desired design.   See the modeling section of this document and your 3D modeling software for more information about creating meshes. 

 

Note:  For best results, even if you are not planning to texture map your object inworld, it is best to apply UVW mapping to all objects.  It is not required that you add a texture to your object.

Applying Textures Overview

You can apply a texture to a model in different ways.  You can either assign the material maps (textures) to a model when your building using your 3D modeling tool, or you can dynamically assign textures using the property pages or via scripting.   

 

For example, if you want texture "RedVelour" to be applied to the mesh "chair seat" and texture "Wood" to be applied to the mesh "chairframe" you must set the meshes and design the UVW mapping in your 3D modeling program.

 

You will need to use your 3D modeling tool to determine how the texture is mapped onto the model.  To design the way the texture is applied to the model, you will need to prepare the mesh for texture mapping using UVW mapping.  You will then need to apply the texture to the mesh. 

You are able to view the texture names and mesh names in Virtual Worlds, so we suggest naming your textures and your meshes appropriately.

 

Note:  No texture is required on a model.

Why do I  not “see myself” in my 3D spaces?

By default, when using the standard client, you views the space in first person point of view (as if “looking from his own eyes”).   To see yourself, change your camera view in the standard client by change to viewing over you shoulder or in front of yourself.

 

Can end users modify or create environments?

Why do the walls seems to overlap and re-draw in the incorrect Z order when I move in the space?

When creating the model of your environment, make sure the polygons in your model do not unnecessarily overlap in the corners.  Also make sure that you do not place planes close to one another.  The redraw errors are caused by Direct 3D Render's Z buffer.

 

To avoid this problem when modeling custom rooms, you can align edges using a 3D snap feature in your modeling program.

 

Why does my model not appear in Virtual World?

If your model does not appear, check the following items to make sure it has been built and/or imported properly. 

1.  The model may be offscreen or the origin point of the model may be offset.  To check this in Virtual Worlds, set the render quality of your model to wireframe using the geometry property page of the object.  If you do not see the object in wireframe, check the object position and origin in your modeling program and re-export.

2.  The model may have no color assigned to the geometry or the color is the same as the background of the scene.  To check this, first try typing in the RGB values for a mesh color using the geometry page in Virtual Worlds.  If this doesn't work, assign a material to the object in your 3D modelign program and re-export.

3.  The model may be facing away from you, and you are viewing through the object.  Move around the object or rotate the object.  Reorient you model in your modeling program and re-export if this is the problem

4.  The model may have vertices but no faces, or no vertices and no faces. You will need to check this in your modeling program and re-export.  This may cause other problems in Virtual Worlds in addition to the rendering of the object.

12.           Simulating 3D with Sprites

What are Sprites?

A sprite is a way to represent highly detailed 3D-style objects in a 3D environment without using large numbers of 3D polygons. In Virtual Worlds, the sprite images are actually textures that always turn to face the camera.  Sprites let you use sprites to simulate a 3D modeled object using by 2D textures.  For example, a tree is difficult to model in 3D.  Instead of modeling, I could use a picture of a tree, and have that image turn to face the user to simulate a 3D tree.   Another common example is a billboard or sign.  I may want the billboard to be easy to read and always face the user--rather than having the image texture mapped onto a flat plane and getting distorted if I am viewing from an oblique angle.  This image can also animate (animated .gif). 

 

Simple sprites, like the ones described above are easy to create.  We also have a mechanism for creating more complex sprites with multiple states.  For example, you can have a sprite with 2 states--depending on your angle away from the sprite; it can show you two different states like a back view and a front view. The Virtual World avatar is an example of a more complex sprite.  The avatar has front, side, and back views.  Depending on the user's viewing angle, different images are shown depending on the number of degrees a user is away from the avatar.  In addition, some of the actions are only seen from certain viewing angles-- so when I smile, I wave only in the front image.

In Virtual Worlds, the sprite images are actually textures mapped onto flat 3D polygons in the 3D environment, which always turn to face the camera. The graphical portion of the sprite contains a series of 2D images, or frames, each representing that sprite from a different viewing perspective. When an avatar is seen from different perspectives in the 3D environment, different 2D images can be swapped in to display the corresponding part of the image.

Creating a Simple Sprite (Single State)

When Virtual Worlds sees an object with a texture map but no geometry, it assumes the designer intended to create a sprite.  You can create a sprite by specifying a texture in the Object Explorer or Texture Property pages and specifying no geometry for the geometry property.  You can also create a sprite using the Geometry Property page by typing in a texture in the file path text field.

 

To make an animating sprite, specifying an animating image in the animated .gif format.

 

Note:  Sprites use images, and sprite images should follow all the same guidelines, including file format limitations as other textures used in world.

Complex Sprites (Multiple Views and Animation Sequences)

–                     Sprite elements

To create a sprite with multiple states, you need to create an animated .gif with all of the images and a text file (called a .spr) with the information for the sprite.  You will also need to use script to manage the display of your sprite.  Note:  You will not be able to customize sprites by typing in the name of the .spr file in the geometry or texture property pages.  For more information on this see the DoGesture method in the Virtual Worlds help documents.

 

Sprites in Virtual Worlds are composed of at least two elements:

·         multi-frame, graphic animated .GIF file, which contains all the images required for the sprite effect (from different angles and/or performing different gestures)

·         .SPR file, a text file which defines which frame is displayed when and for how long. When an avatar uses a gesture, the Virtual Worlds is referencing the .SPR file.

·          

You must place both the .GIF and .SPR files in the same directory in order for Virtual Worlds to read in the .spr file.

–                     Sprites with animation sequences (with specific timing)

A sprite with an animation sequence is a sprite with all the above behavior plus the additional ability to display a predefined set of actions that contain one or more image for predefined periods.   For example, if I want my avatar to wave, I could play frame 1 for 4 seconds, then frame 2 for 4 seconds, and repeat that for 3 times creating a wave animation.  Note: Times are specified in milliseconds, with 1000 milliseconds = 1 second.

 

To create persistent states (for example, if I want my dog object to remember that it was in the sleep gesture state when I logged off and to return my avatar to the sleep gesture when I return) you will need to script the persistent state.  The .spr text file cannot remember the state of your sprite.

–                     Sprites with multiple views

A sprite can also have different views from different view angles.  For example, I may see the front of the person when I'm standing in front of it, the left side when I'm viewing from the left side, and so on.  View angles are specified in degrees, with 360 being the front of the object.

·         Animation Sequences (specifies views, sequences, and timing)

–                     Tips when using Sprites

Below are a few tips to keep in mind when working with sprites.

 

1.  All .SPR files must have at least one gesture. Note:  .gif animations number their frames differently than the .spr layout.  .gif animations begin with image #1 and actions begin with image #0.

2.  Virtual Worlds software running on a 133 Megahertz computer is able to display a frame rate of eight frames per second. 3.  Sprites can have a single image. They do not need to have multiple images.

4.  The angle of view is customizable.  Angle 360 corresponds to the front of the object.  

5.  Avatars are a special kind of sprite.  You can make a custom avatar by customizing the .spr file.  See the section below for more information.

13.           Avatar Accessories

Accessories are artifacts that can be displayed on the front view of an avatar.  Accessories can be used if you want to be able to let your avatar wear a badge, or a hat, that they can put on, and position it on the front of the avatar by dragging the object in the self view window.  If an avatara is wearing an accessory it will appear in the avatar's contents.  By default, when the user first wears and accessory, it will appear in the center of the self  view window.

 

To make any artifact an accessory, turn the IsAccessory property set to True.  To specify an image for your accessory, you can specify an image (typically a .gif) using the AccessoryGraphic proerty

 

Note:  The geometry of the image and the accessory graphic are 2 distinct images.  For example, if I "wear" the sword, it may have a flat image that looks good when composited on my avatar, but if I drop it, it could be a 3D sword or use a texture map other than the accessory graphic image.

For more on accessories, refer to accessories in the Virtual Worlds help documents.

 

–                     Compositing the Accessory on the Avatar

When displaying accessories in world on an avatar, the accessory graphic is composited on the avatar graphic.  Avatar graphics are typically .gif files as they are animated and they require transparency.  This means that the image is at most, a 256 color image, and the accessory will be mapped to the limited set of colors in the avatar palette.

 

The accessory graphic should be no larger than the front image of the avatar.  If you are using the avatar construction kit, this is typically 128x128 pixels.

 

The accessory will not be cut off if it extends beyond the avatar image.  This means that accessories like hats might not work or may require "shorter" avatars--avatars that do not extend to the top of the bitmap that allow room for the hat to be placed

–                     Accessory Palettes

As stated above, the accessory will be mapped to the avatar palette.  If you want a group of avatars graphics to look "good" with an accessory, we suggest you create a common set of colors in all your avatars in the palette for each avatar using a tool like Debabliser.  You should then create all of you accessory graphics using the common subset of colors in the avatar palette.

14.           Authoring Tools

Geometry Editor Overview

The purpose of the geometry editor, accessed from the Authoring client is to assemble and modify objects that have been created in a 3D modeling package.  You cannot use the geometry editor to create new models.  The geometry editor lets you easily scale, rotate, and transform (move) objects.  You can also zoom in and zoom out of the object, turn gravity on/off and turn collision on/off.

 

The Geometry editor has a limited set of features.  Below are some hints and tips for using the Geometry editor:

1.  You cannot Undo using the Geometry editor.  Make sure you save your world before doing any major changes and make sure to lock down objects so you do not accidently move them.

 

2.  You are not able to move an object using local coordinates.  Virtual Worlds uses global coordinates.  This means, you can't "move an object forward 1 unit"--instead you need to specify the global x, y , z  coordinates.

 

3.  Rotations are specified using vectors, not degrees.  When specifying a rotation for an object, you will need to specify the rotation using vectors, not degrees.  Try rotating your object in world to the correct position, and then check the direction coordinates using the Object Explorer. 

 

For more help using the Geometry editor see the Virtual World Help docs.

Boundary editor overview

You can use the boundary editor to create floor/ceiling heights for your 3D spaces, and for creating collision detection (passable/not passable), trip wires, and stairs (or other ground level changes).

 

If the step is <.5 unit, the avatar will "step up."  If the step is >.5, the avatar will collide with the boundary (for example, a low wall).  Below are some hints and tips for using the Boundary editor:

 

When using the boundary editor to create collision detection, make sure to check the boundaries in world from all camera views, including any overhead or over the shoulder camera views.  For example, the avatar is one unit wide.  This means the boundary needs to be about one unit inside the wall geometry so the avatar doesn't "pop through" the boundary when viewing in first person perspective.

 

Make sure to design hallways and stairs wide enough to allow for avatar clearance and navigation space.  We also recommend that you design boundaries so they don't have "corners" that could "block-in" an avatar or create spaces that are difficult to navigate.

 

When you create an object, by default, you automatically create an "auto boundary" around the object.  You can use the boundary editor to add, remove, and modify individual vertices or entire boundaries.

 

Boundaries are not objects.  They are associated with an object.  As you move, scale, and rotate your object, the boundary moves with your object.  Floors and ceiling heights are relative to the origin of the object (not to the global coordinates).  One object can have multiple boundaries

 

To prevent users from “slipping out of the corners of your space, make sure collision detection planes intersect at the corners.”

 

If you are creating your world via script, you can export a boundary for an object to script.  Scripts can affect boundaries.  For example, you can write a script that causes something to happen (like a door to appear) when an avatar moves across a specific edge of a boundary.

 

For more help using the Boundary editor see the Virtual World Help docs.

Appendix B:  3D Studio Maxâ Tips and Tricks

Overview of Creating Custom Models for Objects using 3D Studio Maxâ

Microsoft Virtual Worlds comes equipped with many default models that you can use for your objects in Virtual Worlds. If you want to create your own models, then you have to use a 3-dimensional modeling software program to draft the object in all three dimensions.

A three-dimensional object, which is a model in the graphic artist context, has a center point, also known as a origin point. This origin point is used by the ActiveX portions of Virtual Worlds to center the object upon creation, set the correct approach, rotate correctly, and reference all the interactions of the object with the world correctly. The origin point must be the center of the model (or group of models that compose the final model) for that model to rotate and map correctly in Virtual Worlds.

The 3D modeling tool used here to outline the custom model process is 3D Studio Maxâ, versions 1.2, 2.0 or 2.5.

There are several rules to keep in mind for 3D Studio Maxâ:

·         Avoid using Mirror - this reverses the orientation of the model's X, Y, and Z planes when exported to Virtual Worlds.   If the object is mirrored, mirror the Pivot Point and/or sub mesh back to the correct orientation object so it will move properly when modified in Virtual Worlds.

·         The model (or group of models) exported must have origin points centered in the model.

·         The origin point of models and sub-objects must be set properly to correctly display in Virtual Worlds.  Each mesh of a model must have origin point centered.  The object origin must be re-set if the sub object is moved.

·         Multiple mesh objects must have a common center and orientation to display properly in Virtual Worlds.  If an object and its parts appear misaligned or oriented incorrectly in Virtual Worlds,  this may be the problem.

·         Do not use Reset Transform to center origins; this may cause scaling problems when exported to Virtual Worlds.

·         All objects must have a UVW map applied to display properly in Virtual Worlds.  This includes models that you do not intend to texure map.

·         The UVW map (gizmo) for an object must be reset if the sub-object is moved.

·         Object must be oriented in C positive direction (Y facing north in the top view, where Y and Z are in the positive orientation).

If you are new to 3D Studio Maxâ, try the tutorials included in the product. These can teach you the basics of creating, moving, and texturing objects.

If you are creating compound objects, then group the objects together, and set the origin point for the group to the relative center of the entire group.

There are four requirements for exporting an object from 3D Studio Maxâ into Virtual Worlds:

1.        Reset the origin point to the desired center of the object and center geometry to that point.

2.        Apply a material to the object.

3.        Reset or apply a UVW map to the object.

4.        Export the file from 3D Studio Maxâ.

 

Setting the Origin Point of an Object

If you create simple geometric shapes and build from center, the origin point of the object stays relative to the center of the object created. However, if you create any specialized objects or move a sub-object, the mesh will be displaced from the origin.

When you add an object, Microsoft DirectX (inside of Microsoft Virtual Worlds) check the object for the origin point without intrepreting the modify stack in 3D Studio Maxâ. This means you have to manually reset the origin point BEFORE you export the object from 3D Studio Max®. You should also save a copy of the object in .3DS file format, to preserve the 3D Studio Max® information that is stripped away in the .X file format.

If an object is imported to Virtual Worlds with an origin point that is not dead center, the object's approach distance, rotation, initial placement, and coordinates relative to the world are all interpreted by the offset amount. This makes it really hard to use the object.

The first task towards setting the origin point is to match the origin coordinates with the local coordinates. Do NOT use Reset Transform. This command may cause resizing (scaling) problems for this object later in Virtual Worlds. (Objects may be exported with great size.) If you need more help creating objects in 3D Studio Max®, see the tutorials included with 3D Studio Max®.

To reset the origin point of an object to the relative center of the object in 3D Studio Max®.

1.        Select an object.  (To practice create an object, apply edit mesh and at the sub-object level move the vertices at least half the distance of the total object size.)

2.        Right click on the transform button in the main toolbar to bring up the “Transform Type-in” window.  Write down the coordinates from the “Absolute:World” dialog for the selected object .

3.        Reset the pivot of the selected object using in the “Hierarchy panel”.  This will place the pivot at the position and orientation of the object origin.  (Pivot reset allows you to view the actual origin, and not the “Pivot Point” location, which is only used by 3D Studio Max®.)

4.        Go to the Modify Panel and choose “Sub-Object, vertices”.  Go to the Edit drop down menu and click on “Select All”. Write down the coordinates from the “Absolute:World” dialog for the selected Sub-Object mesh.  Compare these coordinates to the coordinates in step 2.  If they are the same the object origin is correctly set.  If they differ continue with step 5.

5.        Make sure you are at the Sub-Object level with all vertices selected (Modify panel).  In the “Transform Type-in Absolute:World” dialog box, type in the coordinates from step 2.  Click on the high-lighted Sub-Object  button in the Modify panel to deselect Sub-Object.  Type in the coordinates from step 4 to align the object to its original (desired) position.

6.        Reset or apply the UVW mapping modifier.  Note:  All models exported to Virtual Worlds must have one aUVW map. The UVW map stays at the local coordinates until you reset it. 

You are now ready to apply a material to the object.

 

Applying a Material to the Custom Object

Before exporting an object from 3D Studio Max® into Microsoft Virtual Worlds, you must reset the origin point of the object, apply a material to the object, and export the object.

When you apply a material to an object, there are specific settings you can choose to optimize your performance and appearance of the object later in Virtual Worlds.

To apply a material to a custom object in 3D Studio Max®:

1.        Select the object, and open the Material Editor.

2.        Click Get Material.

3.        Click New (or from Scene if that is developed).

4.        Choose Standard (in the Material Editor).

5.        Set Diffuse Color. That is the material or polygon color of the object later in Virtual Worlds.

6.        Click Material Map and click Bitmap. You can choose a bitmap now or just leave this empty. You have to select Bitmap to be able to add textures to the object later in Virtual Worlds.

7.        Set specular color to black (0,0,0 RGB), set shininess and shininess strength to 0.  This will properly set the specular color when viewed with lights in Virtual Worlds.

If you have set the origin point of the object to the center of the object, and you have applied the material to the object, then you are ready to export the object.

 

Exporting a Custom Object

To export the .X format file to your Virtual World we used the exporters 3dsexp.dll and 3dsimp.dll (these are shareware). Another possible file conversion program is Polytrans.

To export a custom object from 3D Studio Max®:

1.        In the top viewport, position the object so the normals (the visible side) are facing north. This means that in the back viewport the object's faces/materials should be visible, and shaded in that view.

2.        The object should be positioned at 0,0,0. If there are many objects that make up the single .X file, group them and set the group to 0,0,0. This insures positioning, approach, and rotation consistancy between objects in Virtual Worlds.

3.        Using the 3DSexp.dll export plug in, (Note:  If this is on your local machine, it should appear  in the directory: ..\3DS\STDPlugins\3DSExp.dll. If you do not have this, you will need to get this shareware).  Choose Export from the File menu. Select the destination folder and file name, then click Save.

4.        The Save Object dialog box appears. Click Advanced and select the option Do not include normal data. The normal data can affect scale adversely in Virtual Worlds. Click OK. Click OK (again, to exit the Save Dialog Box).

5.        Save all the source files (3D Studio Max® files) if you want to edit the data further in 3D Studio Max®.

The same process applies to exporting a whole room object, except that the bottom of the room can be set to 0 on the Z axis (instead of the center) and the X and Y axis should be set to 0. This insures that the room is opened in Virtual Worlds at a viewable position.
