How To: Virtual Worlds Registry Keys

by Peter Wong, Microsoft Research

version 01.00.00.0001, 10/25/99

 

This document illustrates the various keys that Microsoft Virtual Worlds adds to the Microsoft Windows registry database and their respective uses.

 

There are 3 main categories of registry keys that Virtual Worlds uses (actually four but you don't need to worry about the fourth - CLSIDs of library contents).  A typical user will only need to deal with the first two:

 

a) Install paths to locations of files

b) Friendly name to checkpoint file mapping

c) Server and World properties

 

a) This is a static set of keys that is created at install time and lets the system know the main install path, and locations for various content.  Most should be self-explanatory.  You would only need to change these if you want to move locations of any of these files.  Useful if you're changing installation drives or if you want to test with new content without erasing installed content (by temporarily pointing the path to a new location).  These paths are the default paths that the system will search for content on.  The RootURL allows additional content paths to be post-pended to the LocalContentPath setting.

 

Note that AvatarPath and WorldPath define where checkpoint files are located, and not where avatar graphics and world content are located.  Checkpoint files are created locally for worlds running on a local server and avatars that are created on a local client.

 

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\V-Worlds]

"InstallPath"="C:\\Program Files\\Microsoft Virtual Worlds"

 

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\V-Worlds\Paths]

"AvatarPath"="C:\\Program Files\\Microsoft Virtual Worlds\\Avatars\\"

"AvatarGraphicsPath"="C:\\Program Files\\Microsoft Virtual Worlds\\Local Content\\Avatar Graphics\\"

"ContentPath"="C:\\Program Files\\Microsoft Virtual Worlds\\Local Content\\"

"WorldWizPath"="C:\\Program Files\\Microsoft Virtual Worlds\\Local Content\\SDK\\Wizards\\World\\"

"HelpPath"="C:\\Program Files\\Microsoft Virtual Worlds\\docs\\"

"WorldPath"="C:\\Program Files\\Microsoft Virtual Worlds\\Worlds\\"

"DirectoryServiceURL"="LDAP://vworldsint/o=Microsoft/ou=dynamic"

"DirectoryServicePage"="file:\\\\C:\\Program Files\\Microsoft Virtual Worlds\\Local Content\\Client\\html\\dservice.htm"

 

b) These are created at runtime as each world and avatar is created locally.  Each world and avatar stores properties/profiles in a checkpoint file (per world/avatar).  In the case of each avatar, the registry maps a name to a checkpoint.  In the case of each world, the registry maps a GUID to a checkpoint.  A user would only realistically touch these registry entries if they wanted to move an administrative avatar (world creator) from one machine to another to allow wizard privelages from another client.  Manipulating worlds is best done from the Virtual Worlds Server application.

 

[HKEY_CURRENT_USER\Software\Microsoft\V-Worlds\Avatars\aaa]

@="C:\\Program Files\\Microsoft Virtual Worlds\\Avatars\\aaa.vwc"

 

[HKEY_CURRENT_USER\Software\Microsoft\V-Worlds\Worlds\{212256E2-8014-11D3-9A9F-00C04F5C2468}]

"Path"="test1pt1server"

"FriendlyName"="Friendly name here."

"NotInDirectory"="TRUE"

"InfoURL"="TRUE"

 

c) A few out of world properties that the system or server uses:

            isSDK lets the system know if the SDK files are installed

            MaxUsers is the maximum number of users allowed on a server...0 means no limit

            Local Group is the name of an NT local group for authentcation purposes (on a large intranet, where everyone is authenticated against a common domain, you can set a local group on the server machine to limit access to the VWorld server to only members of that group)

            Authentication (0=off/1=on) - NTLM authentication

 

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\V-Worlds]

"isSDK"="1"

"MaxUsers"=dword:00000000

"LocalGroup"=""

"Authentication"="1"
